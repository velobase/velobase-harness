# 第三方集成梳理指南

本文档定义了框架中每个第三方集成的标准梳理流程。所有集成按此流程执行，确保一致性和质量。

## 框架定位

```
开发者：创造（产品想法）+ 判断（评估结果）
AI：功能实现（写代码）
框架：弥补 开发者+AI 到 完整项目 的距离
```

框架做两件事：

1. **指导 AI** — 提供大局观，避免因开发者经验不足导致的架构和安全问题
2. **提供成熟的第三方集成** — 云数据库、认证、收费、邮件等，开发者填密钥即可用

框架不做的事：

- 不做降级策略（由用户 + AI 根据具体产品决定）
- 不做关联页面（由用户 + AI 根据具体产品实现）
- 不做业务逻辑模板（AI 来写）

## 每个集成的标准梳理流程（6 步）

### 第 1 步：选型

明确这个大类具体要支持哪些第三方服务。

需要回答的问题：

- 这个大类下有哪些主流的第三方服务？
- 我们选择支持哪些？为什么？
- 是否需要增加或删除某些三方的支持？
- 不同三方之间是互斥还是共存关系？

输出：一份选型清单，列出支持的三方服务及选择理由。

### 第 2 步：架构设计

研究选定的第三方服务的接入方式，进行软件架构设计，确保架构对功能的良好支持。

需要回答的问题：

- 每个三方的 SDK / API 是什么？当前版本？
- 代码如何组织？（目录结构、文件划分）
- 是否需要抽象层来屏蔽不同三方的差异？（如邮件：Resend 和 SendGrid 用统一接口）
- 数据模型需要什么？（Prisma schema 是否需要新增/修改）
- 与框架其他部分的依赖关系是什么？

输出：架构设计文档，包含目录结构、核心类/函数的职责划分、数据流图。

### 第 3 步：接口定义

明确第三方功能的边界，确定对外的调用接口或交互方式。

需要回答的问题：

- 业务代码调用这个集成的方式是什么？（函数签名、参数类型、返回类型）
- 功能边界在哪里？这个集成做什么、不做什么？
- 接口的粒度是什么？（太细则 AI 需要理解太多细节，太粗则不够灵活）
- 是 server-only 还是也有 client-side 的接口？

输出：TypeScript 接口/类型定义，包含每个对外函数的签名和 JSDoc 说明。

### 第 4 步：配置

梳理该集成需要的所有环境变量和密钥。

需要回答的问题：

- 需要哪些环境变量？（API Key、Secret、Webhook URL 等）
- 哪些是必填、哪些是可选？
- 开发者需要去哪里获取这些密钥？（如：去 Stripe Dashboard 创建 API Key）
- 是否需要在第三方平台做额外配置？（如：设置 Webhook URL、配置 OAuth 回调地址）

输出：在 `.env.example` 中添加相关变量及注释，在集成文档中说明获取步骤。

### 第 5 步：异常处理

设计该集成的异常和边界情况处理方案。

需要回答的问题：

- 第三方 API 调用失败时怎么处理？（超时、限流、服务不可用）
- 是否需要重试？重试策略是什么？（次数、间隔、指数退避）
- Webhook 场景下如何保证幂等？（同一事件重复投递）
- 错误如何分类？（可重试 vs 不可重试 vs 需人工介入）
- 对调用方如何表达错误？（统一的错误类型）

输出：错误处理代码和错误类型定义。

### 第 6 步：AI 引导

设计让 AI 能顺利使用该集成的引导方式。

需要回答的问题：

- AI 使用该集成时最可能犯什么错？如何通过类型系统预防？
- 是否有必填参数可以强制正确用法？（如 storage.upload 必须指定 allowedTypes）
- 是否有默认行为可以自动兜底？（如 mutation 自动要求认证）
- 需要在 AGENTS.md 中添加什么规则？
- 是否需要示例代码让 AI 参考？

引导强度优先级：

1. **类型强制**（最强）— AI 不可能绕过，如必填参数、受限类型
2. **默认行为**（强）— 不需要 AI 做任何事就自动生效，如 middleware
3. **文档规则**（弱）— 告诉 AI 应该怎么做，但 AI 可能忽略

输出：类型约束代码 + AGENTS.md 中的相关规则。

## 集成梳理产物清单

每完成一个集成的梳理，应产出以下内容：

```
docs/integrations/<name>/
├── README.md          # 选型说明 + 架构设计 + 接口文档 + 配置指南
└── DECISIONS.md       # 关键设计决策及理由（可选）

src/capabilities/<name>/        # 或在现有目录中整理
├── index.ts           # 统一导出
├── <provider>.ts      # 各三方 provider 实现
├── types.ts           # 对外类型定义
└── errors.ts          # 错误类型定义

.env.example           # 新增相关环境变量
AGENTS.md              # 新增该集成的 AI 使用规则
```

## 质量检查

每个集成完成梳理后，用以下问题自检：

- 开发者填上密钥后，该功能是否能直接使用？
- AI 只看类型定义和 AGENTS.md，能否正确调用该集成？
- 第三方 API 异常时，是否有合理的错误处理？
- 环境变量是否已添加到 `.env.example` 并有注释说明？
- 代码中是否有类型约束来引导 AI 的正确用法？
